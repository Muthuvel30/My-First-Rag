# My First RAG Project

_A hands-on build to understand how retrieval-augmented generation actually works under the hood._

This is my starting point for actually understanding RAG — not just using it, but seeing what happens inside.

Building it taught me a lot: chunking, embeddings, vector databases, and how similarity search actually works. It's also where I learned Docker properly — why you'd use it, and how to take a project from writing a Dockerfile, to building an image, to running a container. Along the way I got into APIs and, importantly, how to keep an API key secure instead of hardcoding it.

**Live demo:** https://my-first-rag-9uss.onrender.com/docs

---

## Tech Stack

- **Language:** Python 3.12
- **API framework:** FastAPI, Uvicorn, Pydantic
- **Vector database:** ChromaDB
- **AI / models:** Google Gemini API — `gemini-embedding-001` (embeddings), `gemini-2.5-flash` (chat)
- **PDF processing:** pypdf
- **Config / secrets:** python-dotenv (loads `GEMINI_API_KEY` from a local `.env`; read via `os.environ`)
- **Deployment:** Docker (containerization), Render (hosting)

## How to Run

**Live demo:** https://my-first-rag-9uss.onrender.com/docs
Upload a PDF, then ask a question. Note: on the free tier the app sleeps when idle, so the first load may take ~30–60s, and uploads reset when it restarts.

### Run locally

1. Clone the repo:

   ```
   git clone https://github.com/Muthuvel30/My-First-Rag.git
   cd My-First-Rag
   ```

2. Install dependencies:

   ```
   pip install -r requirements.txt
   ```

3. Get a Gemini API key from Google AI Studio, then create a `.env` file in the project root:

   ```
   GEMINI_API_KEY=your_key_here
   ```

4. Run the app:
   ```
   uvicorn app:app --reload
   ```
   Open http://127.0.0.1:8000/docs to test.

### Run with Docker

1. Build the image:

   ```
   docker build -t rag-app .
   ```

2. Run the container:
   ```
   docker run -p 8000:8000 -e GEMINI_API_KEY=your_key_here rag-app
   ```
   (The key is passed in with `-e` because `.env` isn't copied into the image.)

## API Endpoints

### `POST /upload`

Upload a PDF to index it. The file is read, split into chunks, embedded, and stored in ChromaDB.

- **Input:** a PDF file (multipart form-data)
- **Returns:** a confirmation message

### `POST /ask`

Ask a question about the uploaded document(s).

- **Input:** JSON body — `{"text": "your question here"}`
- **Returns:** an answer generated by the LLM from the retrieved chunks

---

## How It Works

1. **`app.py`** — the main file, where all the other modules come together. It exposes the API endpoints. `/upload` takes a PDF as input via an HTTP POST request.

2. **`load.py`** — the uploaded file goes to `load_pdf()`, which converts the raw bytes into readable text, then splits that text into chunks.

3. **`embed.py`** — each chunk is embedded using the Gemini embedding model. Every chunk is stored with a unique id (generated with `uuid4`), its document text, and its embedding values, in a collection called `docs`.

4. **`/ask`** — now we can query. The question is embedded too (no splitting needed since it's short). Then `collection.query()` finds the most similar chunks — I set `n_results=2` to fetch the top 2 matches.

5. **`llm.py`** — the question plus the retrieved chunks are sent to the LLM as a prompt, and it forms an answer from that context. If the answer isn't in the retrieved chunks, it says "I don't know" — because the prompt explicitly tells it to.

> **Tip (especially if you're new to this):** before running the full app, it's worth testing each module on its own — check that `load.py` reads and chunks a PDF, that `embed.py` returns embeddings, and that `llm.py` responds — so if something breaks, you know which piece to look at. Building and testing piece by piece made debugging much easier for me.

## Project Structure

```
├── app.py            # FastAPI app — defines the /upload and /ask endpoints
├── load.py           # reads the PDF (bytes → text) and splits it into chunks
├── embed.py          # embeds chunks/queries using the Gemini embedding model
├── llm.py            # sends the prompt + retrieved chunks to Gemini for the answer
├── requirements.txt  # Python dependencies
├── Dockerfile        # builds the container image
└── .dockerignore     # files kept out of the image (.venv, .env, Chroma_db, etc.)
```

## What I Learned

- **The same embedding model has to be used for both documents and the query.** I learned this the hard way when a question returned "I don't know" — the vectors only line up for similarity search if they're produced by the same model on both sides.
- **Retrieval quality depends heavily on chunking and the prompt, not just the model.** I hit three real failures: a bare name that had no context to embed against, a question whose words didn't match the document's wording, and the LLM copying text verbatim until I fixed the prompt to tell it to summarize.
- **A vector database finds by similarity, not by id.** The id is for identity and overwrites/deletes, not for search — I only understood why ids mattered once I hit the second-upload overwrite problem.
- **Docker: the difference between a Dockerfile, an image, and a container**, and why `RUN` (build time, once) differs from `CMD` (runs when the container starts). I also learned why dependencies are installed before copying the code — Docker caches layers, so it keeps rebuilds fast.
- **"It works" can be a lucky accident.** At one point my embedding function looked fine but was actually broken — it only worked because old correct data was still cached. I learned to test on a clean, freshly-built state, not on leftover data.
- **How to keep an API key secure** — loaded from a `.env` file locally and set as an environment variable on the host, never hardcoded or committed to GitHub.

## Known Limitations & Future Improvements

- **No user separation.** There's a single shared collection, so any visitor can query documents anyone else uploaded, and multiple uploads mix together. A real version would separate documents per user (separate collections or a user id in the metadata) behind some authentication.
- **Storage is not persistent on the free host.** The container's filesystem resets when it restarts or sleeps, so uploaded documents have to be re-uploaded. A persistent disk or an external vector store would fix this.
- **Basic fixed-size chunking.** Chunks are split by a fixed size with overlap, which can cut sentences in half. Smarter, sentence- or section-aware chunking would improve retrieval.
- **Minimal error handling.** The LLM response parsing assumes a well-formed response; it could be made more robust against API errors.
